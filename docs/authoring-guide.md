# 寫作指南

本文件是書稿語言、章節結構、範例、版本與驗證方式的作者規範。讀者可見的
七步證據閉環與內容狀態以[序章](../manuscript/preface.md)為正本；全書
教育目標、章序、系統能力地圖與開發里程碑統一見[課程大綱](curriculum.md)。

## 讀者假設

主要讀者是懂基本電腦操作、可能只接觸過少量 Python，但沒有完整量化交易經驗
的人。正文首次出現專有名詞時先用白話解釋，再給英文名稱與系統型別。

## 事實來源

配套專案發生衝突時，依其 `AGENTS.md` 指定的權威順序判斷：架構設計、工程
總綱、品質政策、開發計畫、當前 Phase 簽名級設計。程式、測試與已發布 tag
用來確認功能是否真正可用；規劃文件中的未勾選項目不能寫成完成式。

## 範例規則

- 範例優先呼叫 production API 或正式 CLI，不複製第二套交易邏輯。
- 會計數字由字串建立 `Decimal`，不以二進位 float 示範金額計算。
- 引擎時間使用注入的 Clock；不得以目前牆鐘製造不可重現結果。
- 每個操作標示工作目錄、前置版本、預期結果與失敗排查方向。
- 外部網路案例與離線固定資料分開，不能以 mock 冒充真實下載成功。
- 不在書籍 repository 存放 API key、帳戶資料或其他秘密。

## 章節狀態與版本

狀態名稱與讀者承諾只在[序章的內容狀態](../manuscript/preface.md#內容狀態)
定義，作者不得另造近義標籤。所有章節與具操作性的 front matter 都在開頭記錄
`配套基線`、`內容狀態` 與 `最後驗證日期`，並以章末作者驗證紀錄支持宣稱。

若配套介面改變，先更新課程大綱的能力地圖，再修訂受影響章節與驗證紀錄。

所有書稿 PR 在交付前由 repository 根目錄執行 `./scripts/book-check`。出版格式、
固定工具版本、孤兒與 metadata 適用範圍統一見[出版工具鏈與 book check](publishing.md)；
不得以單獨執行 renderer 取代完整品質入口。

章末「作者驗證紀錄」是讀者可見摘要；claim 級版本、命令、oracle、結果、差異與
重驗條件統一維護在[作者驗證台帳](verification-ledger.md)。新增或重驗章稿時必須
在同一個 PR 更新台帳，不另建試算表或第二份 Markdown 台帳。讀者完成練習產生的
專業紀錄屬於讀者，不提交進作者台帳。

程式、HTML 或 Markdown 語法示範放在 fenced code block，不直接嵌入可執行的 raw
HTML。書稿中的圖片、script 與其他 reader resource 必須納入 repository 並可由
book check 在離線產物中解析；外部網址只作為讀者連結。

## 數學公式

書面數學關係使用 LaTeX，由固定在 repository 內的 MathJax 轉成 HTML SVG。系統
欄位名稱、命令、程式碼與逐字預期輸出仍使用 inline code 或 fenced code block；
不要把可複製的 evidence output 改成數學排版。

mdBook 來源必須使用經 Markdown escape 的 delimiter：

```markdown
行內公式：\\( Q\_{\mathrm{net}}=Q\_{\mathrm{spot}}+Q\_{\mathrm{perp}} \\)。

\\[
\mathrm{PnL}\_{\mathrm{net}}
=\mathrm{PnL}\_{\mathrm{fill}}+\mathrm{CF}\_{\mathrm{funding}}-\sum\_i F\_i
\\]
```

- 行內公式使用 `\\(` 與 `\\)`，且同一行成對。
- 展示公式使用各自獨立一行的 `\\[` 與 `\\]`。
- 不使用 `$$`；mdBook 的 Markdown 路徑不把它當成受支援的正式 delimiter。
- 多行對齊使用 `aligned`；來源中的 LaTeX 換行要寫成四個反斜線，讓 HTML 保留
  兩個反斜線給 MathJax。
- 變數下標使用 `{}`，且 source 的 `_` 必須寫成 `\_`，避免 CommonMark 在
  MathJax 前把它解析成 emphasis；單位用 `\mathrm{}`，分式用 `\frac{}{}`。
  不要再以 `p_entry`、`USDT/BTC` 或 `/` 的純文字近似正式公式。
- 同一公式首次出現時仍須以正文說明符號、單位、方向與適用前提；排版變漂亮不
  會使公式自動正確。

## 共用版本固定機制

所有操作章共用[實作準備](../manuscript/front-matter/setup.md)建立的版本化 worktree：

- 章首基線必須同時包含 tag 與 commit，例如 `v0.3.0@c999965e5cc9`。
- 不得要求讀者在正在開發的 `../emmet-qt-bt1` 上切換 tag 或 detached HEAD。
- 第一個操作前核對 `git rev-parse HEAD`，並確認 `git status --short` 無輸出。
- 安裝依賴只能使用已提交 lockfile：先 `uv lock --check`，再
  `uv sync --locked --dev`。
- 章末作者驗證紀錄保存完整 SHA、實際命令、結果與已知差異。

配套 tag／commit、正式入口、命令、schema 或預期輸出任一改變，即觸發「需重驗」；
不能因舊章文字仍可閱讀就保留「可操作」。

## 語言與風險

正文使用臺灣繁體中文，保留程式識別字與必要英文術語。區分「模擬結果」、
「歷史結果」與「實盤結果」；避免保證獲利、低風險或可直接上線等措辭。涉及
交易所規則、法規、費率與 API 現況的內容，在出版或發布前必須重新查證。

## 章節固定結構

每章依序章唯一定義的[七步證據閉環](../manuscript/preface.md#七步證據閉環)
安排；下方章節模板是它的作者面實作，不另行改寫閉環定義。

沒有正式入口的章節可使用手算、設計閱讀或已發布測試證據，但必須標示內容
狀態，不得虛構 CLI、API、MCP 工具或輸出。

## 章節模板

新章以以下模板起稿，再依內容刪除不適用的小節。

```markdown
# 章名

> 配套基線：`emmet-qt-bt1 <tag>@<commit>`
> 內容狀態：穩定概念／可操作／概念預覽／規劃中／需重驗
> 最後驗證日期：YYYY-MM-DD

## 學習目標

列出三至五項讀者完成本章後能做到、判斷或解釋的事情。

## 問題情境

從具體問題開始，不先堆疊術語。

## 執行前預測

要求讀者先寫下預期結果與理由。

## 核心概念

用最低完整理論解決問題，首次出現的術語附英文和白話定義。

## 系統對照

指出正式模組、公開介面、資料流與責任邊界；未發布能力明確標示。

## 動手驗證

寫明工作目錄、前置版本、輸入、完整指令或手算步驟，以及預期結果。

## 結果解讀與決定

說明成功、失敗與降級各代表什麼，以及應繼續、停止或修改什麼。

## 常見陷阱

列出資料、時間、數值、成交、風控或操作上的典型錯誤。

## 對系統的回饋

說明本章觀察可以形成哪種資料、文件、測試、策略或系統改善。

## 小結與練習

以可檢查且不需真實資金或私密 API key 的任務收尾。

## 作者驗證紀錄

- 對照 tag／commit：
- 驗證命令：
- 通過結果：
- 待處理差異：
```
