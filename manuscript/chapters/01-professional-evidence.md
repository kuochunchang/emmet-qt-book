# 專業不是預測準，而是能對決定負責

> 配套基線：`emmet-qt-bt1 v0.3.0`
> 內容狀態：可操作
> 最後驗證日期：2026-07-12

## 學習目標

讀完本章後，你應該能夠：

- 用自己的話說明量化交易與一般程式交易的差異；
- 把一個交易想法拆成資料、訊號、訂單、成交、會計與風控；
- 分辨研究結果、回測結果與實盤結果；
- 說明為什麼「會寫策略」不等於「有能力安全地交易」。

## 從一句交易想法開始

假設有人說：「當永續合約比現貨貴太多，我就買現貨、放空永續，等價差縮小。」

這句話聽起來已經是一個策略，實際上卻留下許多問題：

- 「貴太多」是多少？要看價格差、價格比，還是統計上的 z-score？
- 現貨與永續的資料是否來自同一時刻？
- 兩張訂單哪一張先送？如果只成交一邊怎麼辦？
- 永續合約的資金費率會增加收入，還是吃掉利潤？
- 手續費、滑點與最低下單量算進去了嗎？
- 價差沒有縮小反而擴大時，什麼條件下要停止？

量化交易的工作，就是把這些含糊之處變成明確、可執行、可檢查的規則。它不
只是用程式找買賣點，而是建立一條從證據到決策，再從決策到風險控制的完整
鏈路。

## 六個不能省略的環節

一個最小但完整的量化交易流程，可以拆成六個環節。

1. **資料（data）**：我們在什麼時刻知道哪些價格、費率與市場狀態？
2. **訊號（signal）**：資料滿足什麼條件時，策略產生交易意圖？
3. **訂單（order）**：意圖如何變成市場、限價、只減倉或 post-only 訂單？
4. **成交（fill）**：訂單是否真的成交，成交多少，價格與費用是多少？
5. **會計（accounting）**：成交、資金費與手續費如何改變錢包和持倉？
6. **風控（risk control）**：哪些情況要拒單、減倉、停止策略或緊急退出？

少掉任何一環，漂亮的績效都可能只是錯覺。例如，拿收盤價產生訊號，又假設
自己能在同一個收盤價無摩擦成交，就可能使用了當時尚未真正可交易的資訊。
這類錯誤常被稱為未來函數或前視偏誤（look-ahead bias）。

## 這六個環節在系統的哪裡

配套系統沒有把所有事情塞在一支策略腳本裡，而是刻意分開責任。

| 環節 | 系統位置 | 責任 |
|---|---|---|
| 資料 | `quant.data`、`quant.common.datasource` | 取得、驗證、保存並按時間提供資料 |
| 訊號 | `quant.strategy_api`、策略程式 | 根據策略可見資訊產生意圖 |
| 訂單 | `quant.common.models.orders` | 表達訂單內容與生命週期 |
| 成交 | `quant.common.fill`、`quant.common.execution` | 驗證成交資格、價格、數量與路由 |
| 會計 | `quant.common.engine.accounting` | 維護訂單、錢包、持倉和資金預留 |
| 風控 | `quant.common.rules`、Phase 4 引擎風控 | 交易所規則與組合／會話風險限制 |

這個分工非常重要。策略可以提出「我想買」，但它不應該自行宣告「我已經用
理想價格成交」，更不能直接修改帳戶餘額。成交與會計由系統統一處理，才能讓
不同策略在相同假設下被比較，也才能核對每一筆錢去了哪裡。

## 量化不等於預測

有些策略試圖預測下一段價格方向；另一些策略關心的是價差、波動、資金費、
流動性或市場結構。量化方法的核心不是一定要預測成功，而是：

- 把假設寫清楚；
- 使用當時真正可得的資料；
- 用一致規則執行；
- 記錄結果與失敗；
- 在新資料上接受反駁。

因此，一個最後被資料否定的假設，也可能是一項成功的研究。相反地，一個只在
反覆調參後看起來很賺錢、卻無法說明資料邊界與測試過程的策略，不是可靠成果。

## 三種結果不能混為一談

### 研究結果

研究結果回答「這個現象值得繼續調查嗎？」它可能來自統計摘要、圖表或快速
篩選，還沒有完整模擬訂單和帳戶。

### 回測結果

回測結果回答「在指定歷史資料與明示成交假設下，這套規則會產生什麼結果？」
它比研究結果完整，但成交仍是模型，不是交易所當時對真實訂單的承諾。

### 實盤結果

實盤結果來自真實訂單、真實延遲、真實流動性與真實故障。它會受到斷線、拒單、
部分成交、規則改變與人為操作影響。即使回測與 Paper 使用同一策略介面，實盤
仍然有不可消除的額外風險。

所以，「歷史回測年化報酬率是多少」不能直接回答「明年能賺多少」。本書後續
每份結果都要交代資料版本、成交假設、費用、風險限制與尚未模擬的部分。

## 動手閱讀：先從型別與測試認識系統

目前不急著寫策略。先到配套專案觀察系統如何描述訂單：

```bash
cd ../emmet-qt-bt1
sed -n '1,240p' src/quant/common/models/orders.py
```

接著閱讀訂單模型的測試：

```bash
uv run pytest tests/unit/test_models_orders.py -q
```

你不必一次看懂每行 Python。先找出以下資訊：

- 一張訂單有哪些欄位？
- 數量與價格使用什麼數值型別？
- 訂單有哪些狀態？
- 哪些狀態轉換會被拒絕？

預期測試應全部通過。若環境尚未安裝，先不要任意改動依賴；第 4 章會完整建立
Python 3.12 與 uv 環境。

## 常見誤解

**誤解一：策略的勝率高就代表能賺錢。**  盈虧還取決於平均獲利與虧損、費用、
滑點、槓桿和極端損失。

**誤解二：回測有用到真實 K 線，就等於結果真實。**  資料是真實的，不代表
成交假設、訂單容量和當時可得資訊也真實。

**誤解三：市場中性就是沒有風險。**  兩條腿可能失衡，基差可能擴大，資金費率
可能反轉，保證金也可能先耗盡。

**誤解四：自動化會消除人為錯誤。**  自動化只會忠實放大寫進系統的規則；錯誤
的資料、假設、權限或風控也會被自動執行。

## 小結與練習

量化交易不是一條預測公式，而是一套可檢查的決策與執行流程。配套系統用模組
邊界把資料、策略、成交、會計和風控分離，讓每一步都能被測試與重播。

請用不超過 100 字寫下一個交易想法，再把它拆成六個環節。若其中任何一格只能
寫「到時候看情況」，那正是下一步需要定義與驗證的地方。
