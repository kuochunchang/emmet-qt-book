# 從研究到實盤：能力地圖與晉級關卡

> 配套基線：`emmet-qt-bt1 v0.3.0@c999965e5cc9`
> 內容狀態：穩定概念
> 最後驗證日期：2026-07-14

## 學習目標

讀完本章後，你應該能夠：

- 依序區分探索、回測、穩健性、Paper、Testnet 與 Live；
- 說明每一階段能支持什麼結論，以及不能證明什麼；
- 為晉級寫出進入條件、停止條件與明確的 no-go；
- 用配套版本、內容狀態與正式入口判斷能力是否真的交付；
- 留下一份個人學習路線與風險承諾，不把「想試試看」當成跳級理由。

## 問題情境：看到目錄，不等於拿到產品

你在 `v0.3.0` 的設計文件裡看見完整工作流程，也在原始碼樹裡看見
`quant.backtest`、`quant.mcp`、`quant.paper`、`quant.live` 與
`quant.strategy_api`。LLM 因此說：

> 套件名稱和工具設計都在，應該已經可以直接跑回測，再接 Testnet；如果結果好，
> 就可以小額實盤。

這個推論把三件不同的事混在一起：設計已寫下、程式名稱已預留，以及讀者可用的
正式入口已發布。它還跳過穩健性與 Paper，並把 Testnet 當成績效證據。

本章不啟動回測、Paper、Testnet 或 Live，也不提供尚未發布能力的假想指令。我們
只做紙筆晉級稽核：先判斷每一階段需要什麼證據，再判斷目前版本交付到哪裡。

## 執行前預測

先不要看後面的判定。針對章首情境，寫下你的預測：

1. 原始碼中有同名 package，是否足以證明功能可操作？
2. 回測結果很好，是否可以略過穩健性與 Paper？
3. Testnet 成交與 PnL，是否可以支持策略晉級？
4. 哪三項證據同時成立時，你才會把一項能力標成「已交付」？

最後先選一個暫定決定：繼續找證據、停止、退回前一階段，或 no-go。不要把
「先執行看看」當成第五個答案。

## 核心概念：每一關只回答有限問題

晉級不是把同一個績效分數越算越精確。每一階段都在排除不同類型的錯誤，而且
前一關的綠燈不能代替下一關的證據。

```text
探索
  → 回測
  → 穩健性與壓力檢查
  → Paper local
  → Testnet 鏈路冒煙
  → 小額 Live 候選
  → 只有新證據支持時才考慮擴大

任一關證據不足或風險超標
  └──► 停止、退回修改，或明確 no-go
```

### 六階段證據地圖

| 階段 | 能支持的結論 | 不能證明 | 進入條件 | 停止／退回條件 |
|---|---|---|---|---|
| 探索 | 現象、資料範圍與可能機制值得形成可否證假設 | 策略可執行、可獲利或已通過風控 | 問題、觀察量、資料來源與失效條件說得清楚 | 資料時點不明、樣本不足，或只能事後解釋 |
| 回測 | 固定歷史資料與明示模型下的訂單、會計、成本與風險軌跡可重現 | 未來績效、真實成交、真實延遲或交易所接受訂單 | 假設、資料、策略、成本與成交模型已固定 | 會計不守恆、資料穿越時間、結果無法重現或成本後失效 |
| 穩健性與壓力 | OOS／Walk-Forward、樣本與參數敏感度可支持繼續或淘汰；壓力情境可檢查風控行為 | 真實成交品質、即時鏈路可靠性，或壓力情境本身的 alpha | 回測與所有嘗試完整留存，判準在看結果前固定 | OOS 衰退、低樣本、過度擬合，或風控動作不符 oracle |
| Paper local | 主網即時資料、會話、恢復與本地模擬成交可揭露工程與行為差異 | 交易所排隊、真實滑點、真實強平／ADL、真實 API 錯誤或實盤獲利 | 穩健性通過；版本化策略、runbook、監控與停止條件齊全 | 斷線／恢復失敗、風險超標，或與回測差異無法解釋 |
| Testnet | 鑑權、下單、撤單、回報與對帳鏈路冒煙 | 策略績效；Testnet 成交價、持倉與 PnL 都不能當晉級證據 | Paper 證據足夠，且有隔離的 Testnet key 與清理程序 | 鏈路、冪等、回報或對帳失敗 |
| Live | 該版本、規模與期間的實際執行與運維證據 | 未來獲利、放大後仍安全，或風險已消除 | 正式入口已發布；前序證據、帶外授權、硬限制、對帳、急停與稽核有效 | 任一前提失效、外部持倉不明、對帳不一致、超限或人類作出 no-go |

這張表刻意讓「不能證明」與「能支持」並列。回測的真實範圍是「歷史資料加
模型」；Paper 能提早暴露即時資料與恢復問題；Testnet 只負責把 API 鏈路問題
留在沒有真實資金損失的地方。

## 晉級關卡不是單向電扶梯

每一關都必須允許三種專業結果：

- **晉級**：證據達到事先寫下的 oracle，且下一階段的新風險已有控制；
- **退回**：問題可以透過修正假設、資料、策略、風控或程序再驗證；
- **no-go**：缺口不可接受、成本不值得，或風險超出目前能力與預算。

no-go 是「在目前證據下不繼續」的正式決定，必須記錄觸發條件與重啟研究所需
的新證據。相反地，「結果不錯，所以繼續」沒有 oracle，也沒有說明新增風險，
不構成晉級決定。

參數、策略程式、資料、成交模型、風控設定或配套版本只要改變，先前證據就可能
不再適用。必須指出受影響的關卡並重驗，不能沿用另一個策略雜湊、資料或版本的
綠燈。

## 系統對照：設計路線與已發布能力分開看

配套系統的權威設計採漸進式驗證：回測與 Walk-Forward 之後先過穩健性與壓力
檢查，再進 Paper、Testnet 冒煙與小額 Live。設計也明確限制證據能力：Paper 用來
驗證策略邏輯與工程鏈路，Testnet 的成交價、持倉與 PnL 不作決策證據，小額實盤
是否執行仍由人類裁決。

但這是完整系統的設計路線，不是 `v0.3.0` 的操作清單。固定在本章基線檢查完整
source tree 時，下列五個未來 package 都只有各自的 `__init__.py`：

```text
src/quant/backtest/__init__.py
src/quant/live/__init__.py
src/quant/mcp/__init__.py
src/quant/paper/__init__.py
src/quant/strategy_api/__init__.py
```

開發計畫把完整引擎／策略介面、Backtest／MCP、Paper、Live 分別放在後續 Phase。
因此本章可以教你判讀路線與停止條件，卻不能聲稱上述 server、工具或策略入口
已經交付。package shell 是預留結構，不是產品入口；設計中的工具名稱也不是
讀者現在可以執行的命令。

## 三項交付檢查

判斷任何操作能力時，同時檢查以下三項：

1. **配套版本**：章首是否固定到已發布的 `tag@commit`，而不是開發分支、未合併
   PR 或只有名稱相同的目錄？
2. **內容狀態**：章首是「可操作」，還是只承諾「穩定概念／概念預覽／規劃中／
   需重驗」？狀態決定讀者可以依賴到哪裡。
3. **正式入口**：該版本是否真的發布核准的 CLI、API 或 MCP 工具，而且章中命令、
   輸出與作者驗證紀錄已實跑？package、類別名稱、測試 helper 或設計表格都不能
   單獨冒充正式入口。

```text
已發布 tag@commit
＋ 內容狀態為可操作
＋ 正式入口與章中結果已重現
＝ 本章宣稱範圍內已交付

任一項缺少
＝ 不建立操作路徑；改讀概念、保留待驗證問題，或停止
```

「可操作」不是對整個系統的總授權。某個資料 CLI 已發布，不代表同版本的
Backtest、MCP 或 Live 也已發布；必須逐項對照入口與 claim。

## 動手驗證：判讀章首情境

用三項交付檢查重新稽核章首情境：

| 檢查 | 目前證據 | 判定 |
|---|---|---|
| 配套版本 | 本章固定 `v0.3.0@c999965e5cc923281541409cda9502beb93b8a60` | 版本身分明確，但只回答正在看哪個版本 |
| 內容狀態 | 本章是「穩定概念」 | 可以依賴能力地圖與判讀方法，不能執行未發布入口 |
| 正式入口 | 五個相關 package 的完整 tree 都只有 `__init__.py`；設計工具位於後續 Phase | Backtest／MCP／Paper／Live／完整策略介面尚未由本基線交付 |

驗收 oracle 是：

```text
不把 package shell 或設計文件當成正式入口
＋ 不虛構指令、輸出或通過紀錄
＋ 保留探索 → 回測 → 穩健性 → Paper → Testnet → Live 的順序
＋ Testnet 數字不進績效決策
＋ 決定寫成等待相符 release 並重驗，不是先跑看看
```

依這個 oracle，章首 LLM 的建議不通過。正確結果是停止建立操作路徑，而不是在
原始碼裡猜一個可能存在的函式。

## 證據顯示什麼

把實際判定與你的預測相比，應得到四個結論：

- 版本固定只能避免看錯版本，不能自行證明所有設計能力已完成；
- 內容狀態是讀者承諾，「穩定概念」不能解讀成「可操作」；
- 正式入口要由已發布程式與實跑證據共同支持，名稱或空 package 不夠；
- 即使未來入口發布，六階段仍各有證據邊界，不能因工具可呼叫就跳級。

能力地圖同時管理兩種風險：研究過程跳過必要證據，以及教材把未發布設計寫成
現成產品。

## 結果解讀與決定

| 結果 | 何時使用 | 應保存的理由 |
|---|---|---|
| 繼續本階段 | 入口與前置條件有效，但證據仍在形成 | 尚缺哪些 oracle、何時重查 |
| 晉級 | 本階段 oracle 通過，下一階段控制已就緒 | 版本、策略／資料身分、通過證據與新增風險 |
| 退回 | finding 可透過修正後重新驗證 | 退回哪一關、哪些舊證據失效 |
| no-go | 能力未發布、核心證據不足、風險超標或不值得繼續 | 停止範圍、觸發條件與重啟所需證據 |

本章基線對 Backtest／MCP／Paper／Live 的決定是 no-go for operation：可以學習
概念與設計邊界，但不能建立讀者操作步驟。正式 release 發布後仍要逐章重驗，
不能把今天的概念驗證自動升格。

## 常見陷阱

**陷阱一：原始碼有目錄，所以功能只差一條指令。** 空 package、介面草稿或開發
分支都可能存在；只有發布版本的正式入口與實跑證據能支持可操作宣稱。

**陷阱二：回測用了真實歷史資料，所以等同真實結果。** 回測仍受資料可得性、
成交模型、成本與參數選擇約束，不能證明未來或真實成交。

**陷阱三：Paper 跑得久，就等於交易所會照樣成交。** Paper 能檢查即時邏輯與
恢復，但本地撮合沒有真實排隊、深度消耗與全部交易所故障。

**陷阱四：Testnet 是交易所環境，所以 PnL 比回測可靠。** Testnet 市場可被任意
操作且流動性失真；它的價、倉與 PnL 不作績效證據。

**陷阱五：小額 Live 沒出事，就可以直接放大。** 一段期間的成功只支持當時規模
與條件；放大會改變流動性、滑點、保證金與事故後果，必須重新授權與評估。

**陷阱六：no-go 代表研究者不夠積極。** 在能力未發布或證據不足時停止，正是
保護資金、時間與證據品質的專業行為。

## 對系統的回饋

如果三項交付檢查無法完成，缺口可能指出：

- release notes 沒有列出正式入口與不支援範圍；
- package shell 太像已交付能力，缺少清楚狀態說明；
- 報告沒有保存資料、策略、引擎版本或晉級 oracle；
- Paper／Testnet 結果沒有強制標示證據限制；
- 教材的內容狀態、命令與作者台帳互相不一致。

把缺少的版本、入口、schema、拒絕理由與重驗觸發條件寫成 finding。指出哪一項
檢查失敗，以及什麼證據會讓它通過，比籠統要求補文件更容易形成可驗收改進。

## 小結與練習

探索、回測、穩健性、Paper、Testnet 與 Live 是六種證據能力。每一關只能回答
有限問題，並且都允許停止或 no-go。判斷系統能力時，再用版本、內容狀態與正式
入口把設計存在和產品已交付分開。

請為下列四句話各選繼續蒐證、晉級、退回或 no-go，並指出 oracle：

1. 設計文件列出 `run_backtest`，所以本版可以操作。
2. OOS 很好，不必記錄調過多少組參數。
3. Testnet PnL 為正，可以進小額 Live。
4. 新 release 改了結果 schema，但舊章仍看得懂，所以不用重驗。
