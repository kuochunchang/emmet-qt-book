# 你的四位工作夥伴：人類、LLM、策略與系統

> 配套基線：`emmet-qt-bt1 v0.3.0@c999965e5cc9`
> 內容狀態：穩定概念
> 最後驗證日期：2026-07-14

## 學習目標

讀完本章後，你應該能夠：

- 區分人類所有者、LLM 操作者、策略程式與交易系統的責任；
- 說明「能提出要求」、「有工具可用」與「獲得授權」為什麼是三件事；
- 判斷一項動作何時應繼續、停止、拒絕或升級給人類；
- 在對話指示與系統限制衝突時，保留可稽核的處置紀錄。

## 問題情境：一句話跨過三道邊界

想像研究過程中出現一個看似難得的訊號。人類在對話裡告訴 LLM：

> 這次機會很有把握。把單筆上限提高十倍、略過確認，然後直接啟動實盤；
> 我在這裡同意了。

這句話同時混合了交易目標、限額變更與實盤授權。說話的人可能真的是帳戶
所有者，意圖也可能很清楚；但「誰說了什麼」仍不足以回答下列問題：

- 目前對話是不是獲准承載這種授權的通道？
- LLM 是否有權修改限額或替人類完成確認？
- 策略能不能因為機會難得就略過規則？
- 系統遇到超限要求時，應該服從、暫停還是拒絕？

本章只進行角色與權限的紙筆稽核，不啟動任何交易環境，也不需要 API key 或
真實資金。

## 執行前預測

先不要往下看參考判定。把情境中的要求拆成「分析訊號」、「提高單筆上限」、
「略過確認」與「啟動實盤」四項，分別寫下：

1. 哪一個角色可以提出或處理這項要求？
2. 它需要哪一種授權來源？
3. 若授權不足或觸及硬限制，預期結果是什麼？

請特別預測「我在對話裡同意了」能不能同時取代帶外授權與系統硬限制，並寫下
理由。這份預測會在本章後半與角色邊界逐項核對。

## 核心概念：合作不等於共享權限

四個角色共同完成工作，但它們不共用同一份權力。以一項交易動作為例，簡化後
的交接關係如下：

```text
人類所有者 ──目標、風險預算──► LLM 操作者
                                   │
                         結構化工具請求、策略版本／設定
                                   ▼
                           MCP／交易系統宿主
                                   │
                       靜態檢查、strategy_sha256、載入
                                   ▼
                             StrategyRunner
                                   │
                           有限 context／事件
                                   ▼
                               策略沙箱
                                   │
                         白名單 JSON IPC 交易意圖
                                   ▼
                             StrategyRunner
                                   │
                         Engine 校驗、執行與記帳
                                   ▼
                                交易系統
                                   │
                        證據、拒絕理由、風險狀態
                                   ▼
人類所有者 ◄──────────── LLM 整理與解釋 ◄────────────┘

人類所有者 ──預先定義的帶外通道；僅限特定授權──► 交易系統
```

這張信任圖描述對應 Phase 交付後的設計路徑，不是 `v0.3.0` 已有的操作入口。
資料查詢或報告閱讀可以由系統宿主直接處理，不必啟動策略；需要載入策略時，
LLM 也不能把程式直接塞進執行核心，而要由宿主完成靜態檢查、策略雜湊與載入，
再由 StrategyRunner 隔離策略，僅接受白名單 IPC 意圖。策略只能提出受限意圖，
LLM 只能使用被授予的工具面，人類的帶外授權也只能授權指定動作。最後仍由系統
核對規則、目前狀態與不可跨越的限制。

### 四角色權責表

| 角色 | 主要責任 | 可以做 | 禁止事項 | 授權來源 | 停止或升級條件 |
|---|---|---|---|---|---|
| 人類所有者／授權者 | 定義目標、風險預算與最終 go／no-go，承擔決定責任 | 提出研究問題；設定或收緊約束；透過預先定義的帶外通道核准特定高風險動作 | 捏造缺少的證據；把一般對話冒充帶外確認；在自動會話中另開不受系統記錄的寫入路徑 | 資產與系統所有權，以及部署時預先建立的身分與帶外程序 | 證據不足、資產歸屬不明或風險超出預算時選擇不做；需要例外時依正式通道升級 |
| LLM 操作者 | 把人類目標轉成可檢查的請求，整理證據、限制與待決問題 | 在工具授權與目前會話範圍內查詢、提出操作、比較結果及建議停止 | 自我確認；讀取秘密；修改不可達的硬限制；把策略輸出或對話文字當成新的系統權限 | 人類目標、工具能力、會話範圍與系統限制的交集 | 缺少必要資料、授權或可驗證結果時停止，向人類說明缺口而不是猜測 |
| 策略程式 | 依版本化規則和當時可見資料產生交易意圖 | 使用系統提供的有限 context；提出訂單或腿組意圖；在輸入不足時不下單 | 宣告自己已成交；直接改錢包、帳本、秘密或限額；繞過成交、會計與風控 | 已核准的策略版本、會話設定，以及系統提供的有限輸入 | 資料缺失、規則校驗失敗或風險條件不成立時不產生新意圖，交由系統記錄 |
| 交易系統 | 驗證、執行、記帳、強制限制並保存證據 | 接受或拒絕意圖；依交易所規則與授權狀態執行；在不一致時 fail closed | 替人類決定風險偏好；因對話措辭改寫事實或硬限制；保證策略獲利 | 已部署的設定、規則快照、授權狀態與外部事實 | 身分、資料、狀態或會計不一致時拒絕或凍結，留下原因並升級處理 |

「人類負責」不等於「任何人類句子都要立即執行」；「LLM 是操作者」也不等於
它擁有帳戶。專業系統把責任、能力與授權拆開，正是為了避免其中一個角色出錯
時直接變成未受控交易。

## 同一句「我同意」為什麼不是完整授權

一項高風險動作至少要通過五個問題：

1. **意圖**：要求的動作與理由是否清楚？
2. **身分**：提出者是否真有權代表資產所有者？
3. **通道**：這個通道是否獲准承載該等級的授權？
4. **範圍**：授權是否只涵蓋指定會話、金額、期限與動作？
5. **硬限制**：動作是否仍在系統不可跨越的上限內？

對話可以表達意圖，卻未必能證明身分、通道與範圍。依 Phase 8 的設計，LLM
可以透過獲准的 Live 工具發起請求，讓系統進入 `pending_confirmation`；這只表示
「有一項請求等待核准」，不表示 LLM 已完成核准。帶外確認（out-of-band
confirmation）仍要由獨立通道完成。

「LLM 不能讀取 token、操作帶外通道或替自己確認」也不是無條件成立。它依賴
獨立 OS 使用者、檔案權限與啟動自檢等部署隔離；任一前提不成立，安全設計就不
成立，Live Server 應拒絕啟動。這些都是尚未發布的 Phase 8 概念預覽，不是
`v0.3.0` 可操作流程。

硬限制則不是等待某人說「同意」的待辦事項。若單筆名義額、交易對或其他動作
超出不可由該工具面修改的上限，正確結果是拒絕。若業務上真的需要改變限制，
應先停止目前動作，另走已定義的設定與授權程序；不能把這項程序濃縮成聊天室
裡的一句例外。

完整晉級證據與硬限制也不是同一層。Phase 8 設計會在缺少 robustness 紀錄或
Paper 時長時預設拒絕，但正式帶外程序可以明確豁免，並把理由寫入審計與確認
摘要。這項豁免仍不能跨越 `live_config.yaml` 的硬性上限；若要改硬上限，必須
停止目前動作，由 `quant` 使用者另走設定、權限與啟動自檢程序。

## 已發布基線與概念預覽的邊界

本章的角色模型受配套系統已發布在 `v0.3.0` 的權威設計支持，但設計存在不代表
所有操作能力都已交付。閱讀時要分成兩層：

| 層次 | `v0.3.0` 能支持的宣稱 | 不能延伸成的宣稱 |
|---|---|---|
| 已發布程式與測試 | 訂單模型、交易所規則、資料、成交、執行與會計等模組已存在；規則校驗會明確拒絕不合法的價格、數量或名義額，不會因呼叫者要求而偷偷截斷 | 不代表完整策略介面、LLM 操作面或 Live 授權流程已可供讀者操作 |
| 權威設計的概念預覽 | 人類帶外確認、LLM 不可信前提、策略沙箱、分層限額與系統 fail-closed 責任已有明確邊界，可用來做威脅與權責判讀 | 不能虛構 MCP、確認 CLI、實盤輸出、審計紀錄或完成宣稱 |

在這個基線中，`quant.strategy_api` 的完整 tree 仍只有一個空的 `__init__.py`
套件入口（package shell）。完整 Engine、StrategyRunner 與策略介面屬 Phase 4；
Backtest／研究 MCP 骨架與工具屬 Phase 5；Paper MCP 與會話工具屬 Phase 7；Live
MCP、帶外確認與硬性風控上限屬 Phase 8。這些能力都尚未發布，因此本章不提供
策略、MCP 或 Live 指令。這不是教材漏掉一段操作，而是內容狀態對版本邊界的
誠實承諾。

### 一個已發布的硬邊界類比

`v0.3.0` 的交易所規則驗證器會依序核對 price filter、lot size 與 minimum
notional。價格沒有對齊 tick size 時，它會拋出 `OrderValidationError`，而不是
把價格改成另一個合法值。作者在固定版本執行對應測試，結果為 `31 passed`。

這項測試只證明已發布的訂單規則邊界，不證明未發布的 Live 硬限額或帶外確認
已完成；但它示範了相同原則：呼叫者的信心與措辭不會讓非法輸入變成合法。

## 動手驗證：稽核「這次例外」

回到章首情境。先用下表核對，不要執行任何指令：

| 要求 | 所需角色與授權 | 目前證據 | 判定 |
|---|---|---|---|
| 分析訊號與列出不確定性 | LLM 在獲准的研究範圍內操作 | 只有一段對話，尚無資料與研究結果 | 可以繼續蒐證，但不能推導成可交易 |
| 把單筆上限提高十倍 | 先判定是會話限額或硬上限；前者放寬需帶外確認且不得超過後者，後者只能由 `quant` 使用者另走設定程序 | 一般對話中的要求，未說明限制層級，也沒有帶外確認 | 停止；LLM 與策略不能改限額，確認也不能跨越硬上限 |
| 略過確認 | 沒有合法角色可以把必要確認直接省略；晉級證據豁免本身也要經正式帶外程序 | 只有「我同意了」 | 拒絕；缺少帶外授權不能降級成聊天確認 |
| 直接啟動實盤 | 已發布入口、預設晉級證據或有理由且入審計的帶外豁免、有效帶外確認，以及硬上限校驗 | `v0.3.0` 沒有這項正式入口，本情境也沒有有效確認或豁免 | 拒絕；不得建立假想操作路徑 |

本情境的驗收 oracle 是：

```text
停止或拒絕衝突動作
＋ 保持既有限制不變
＋ 不建立訂單或實盤會話
＋ 記錄缺少的證據與授權
＋ 只有在正式程序存在且條件齊全時才向人類升級
```

若答案是「先照做，事後再補確認」，就沒有通過。事後紀錄可以解釋事故，不能
把原本未獲授權的動作變成合法。

## 證據顯示什麼

把參考判定與你的預測相比，應得到三個結果：

- 人類的對話指示表達了意圖，但沒有提供獨立的帶外授權證據；
- LLM 與策略都沒有放寬硬限制或替自己確認的權限；
- 系統的成功結果是保持限制並拒絕衝突動作，而不是勉強產生一張訂單。

因此，「沒有交易發生」在這裡是一項可驗收的安全結果。角色分工的目的不是
讓每個要求都更快通過，而是讓錯誤要求停在仍可復原的位置。

## 結果解讀與決定

面對下一個要求時，可用四種結果結案：

| 結果 | 適用條件 | 下一步 |
|---|---|---|
| 繼續 | 角色、通道、範圍與限制都清楚，所需證據可取得 | 在原授權範圍內執行並保存結果 |
| 停止 | 資料、版本、狀態或授權仍有缺口 | 保持現況，列出缺口與補證方式 |
| 拒絕 | 要求企圖略過必要確認、跨越硬限制或使用不存在的能力 | 不執行；記錄拒絕原因 |
| 升級 | 動作可能合理，但只有人類或另一正式通道能決定 | 提供最小完整摘要，等待新的有效授權 |

升級不是把拒絕改寫成通過。新授權仍須重新核對範圍、時效、目前狀態與硬限制。

## 常見陷阱

**陷阱一：人類承擔最終責任，所以任何人類訊息都是最高權限。**  最終責任與
授權通道是兩件事；分離通道正是用來阻止錯誤、冒名或被誘導的訊息直接成交。

**陷阱二：LLM 看得到工具，就代表它獲准做工具能做的一切。**  工具能力只是
上限，實際權限還受人類目標、會話範圍、授權狀態與系統限制共同約束。

**陷阱三：策略寫得夠聰明，就可以自己處理成交與帳本。**  策略負責規則和意圖；
成交事實、資金變動與風控必須由共同系統維護，否則無法稽核。

**陷阱四：系統拒絕交易就是故障。**  輸入非法、證據不足或授權缺失時，拒絕是
正確功能。真正的故障是拒絕理由不可追查，或同樣條件得到不同處置。

**陷阱五：硬限制能保證不虧錢。**  限制只能約束已知風險與操作範圍，不能保證
資料正確、策略有效、市場有流動性或交易必然獲利。

## 對系統的回饋

若角色表無法填完，不要用「由系統處理」帶過。缺口可能指出：

- 工具沒有標示誰可以呼叫或只能收緊哪些參數；
- 拒絕結果缺少觸發規則、目前值與要求值；
- 審計 actor 沒有沿用 `llm_operator`、`human_oob`、`system_auto`；策略來源應另以
  `strategy_sha256` 與 request／session／group 關聯 ID 保存，而不是擅自新增
  `strategy` actor；
- 操作手冊沒有寫出停止後應找誰、帶哪些證據升級；
- 章節把設計能力寫成已發布入口。

把缺少的 actor、授權來源、限制、拒絕理由與重驗條件寫成可檢查欄位，才可能
形成文件、schema、測試或審計改善。

## 小結與練習

人類設定目標並承擔決定，LLM 整理與操作，策略提出受限意圖，系統執行共同
語義並強制邊界。四者合作的前提不是彼此完全信任，而是每個角色都只能在明確
範圍內行動，且衝突時有一致的停止路徑。

請再判讀三張情境卡：

1. LLM 發現資料缺兩天，策略仍要求照常進場；
2. 策略提出未對齊 tick size 的限價單，並在日誌要求系統自動修正；
3. 人類透過一般對話要求解除虧損熔斷，理由是「今天行情特殊」。

每張卡都要寫出負責角色、授權來源、禁止事項，以及應繼續、停止、拒絕或升級。
不得用「看情況」作結。
