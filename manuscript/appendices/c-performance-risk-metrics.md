# C. 績效與風險指標

本附錄目前只收錄[第 6 章](../chapters/06-costs-breakeven.md)直接需要的勝率、賠率、
期望值、換手成本、流動性與容量最小查表。權益曲線、回撤、波動率、Sharpe、
Sortino、VaR 類指標及其限制，會隨對應 active 正文補寫。

正文必須自行說明成本如何讓策略失效；本頁不能取代逐筆資產流、成本表或
continue／no-go 決定。

## 一個 outcome 的邊界

先固定「一筆」是什麼。本附錄的 outcome 是一個完整往返：由沒有該部位開始，
進場後再全部出場，並已包含該往返的所有明示成本。若研究把部分成交、隔夜持有
或多腿組合當成一筆，必須另外定義，不能沿用這裡的數字。

| 符號 | 意義 | 單位／規則 |
|---|---|---|
| `p` | 勝率 | `0 <= p <= 1` |
| `W_g` | 毛獲利 outcome | quote；正數 |
| `L_g` | 毛虧損 outcome | quote；負數 |
| `K` | 每個完整往返的明示總成本 | quote；非負 |
| `W_n` | 淨獲利 outcome | `W_g - K` |
| `L_n` | 淨虧損 outcome | `L_g - K` |
| `EV` | 每個 outcome 的期望值 | quote/outcome |

若不同 outcome 的數量、成交方式或持有期會改變成本，就要為每個 outcome 使用
自己的 `K_i`；不能用平均成本掩蓋尾端大額交易。

## 勝率與賠率

毛賠率在本附錄定義為平均毛獲利絕對值除以平均毛虧損絕對值：

```text
payoff ratio_gross = W_g / |L_g|
```

勝率只回答「正 outcome 的比例」，賠率回答平均贏一次相對平均輸一次的大小。
兩者都沒有自動包含成本；名稱若不標 gross／net，就不足以接受審核。

淨賠率為：

```text
payoff ratio_net = W_n / |L_n|
```

只有 `W_n > 0`、`L_n < 0` 時，這個寫法才保有「贏／輸」的直覺。若成本使原本的
毛贏家變成淨虧損，就應重新分類 outcome，而不是硬算一個看似正常的比率。

## 毛期望值與淨期望值

固定兩種 outcome 時：

```text
EV_gross = p × W_g + (1-p) × L_g

W_n = W_g - K
L_n = L_g - K

EV_net = p × W_n + (1-p) × L_n
       = EV_gross - K
```

最後一條等式是「每個完整往返的成本只扣一次」oracle。若 `W_n`、`L_n` 已扣
成本，又在 `EV_net` 後面扣 `K`，就是雙扣；若只從贏家扣成本，則會高估結果。

第 6 章固定案例：

```text
p = 0.55
W_g = +120.00 USDT
L_g = -80.00 USDT
K = 40.169910 USDT

EV_gross = 30.000000 USDT
W_n = 79.830090 USDT
L_n = -120.169910 USDT
EV_net = -10.169910 USDT
```

這個例子說明高於一半的勝率與 `1.5` 毛賠率仍可能得到負淨期望值。

## 損益兩平勝率

在 `W_n > 0`、`L_n < 0` 且 outcome 只有這兩類時：

```text
p_BE = |L_n| / (W_n + |L_n|)
```

第 6 章得到 `60.084955%`。這只對固定 outcome 與固定成本分布成立；它不是信賴
區間，也沒有證明未來勝率能達到門檻。樣本外、穩健性與統計不確定性屬後續章節。

## 換手與成本負載

第 6 章把單輪換手率定義為：

```text
T = (entry notional + exit notional) / initial equity
```

最低查表應同時保存：

| 欄位 | 為什麼需要 |
|---|---|
| 分子包含單邊或雙邊 | 避免同名換手率差一倍 |
| 分母是初始、平均或期末權益 | 讓不同報告可判斷能否比較 |
| outcome 數量與期間 | 讓每輪成本能合理累積 |
| gross PnL、成本、net PnL | 防止只報費率而漏掉交易活動 |

換手率本身不是好壞結論；只有與費率、spread、滑點、規模及淨 outcome 一起看，
才能判斷高換手是否已吞掉優勢。

## 流動性與容量的最小情境

若只有教學輸入，可使用下式建立 fail-closed 的規模上限：

```text
capacity_qty = scenario_depth × participation_cap
```

| 欄位 | 必須記錄 |
|---|---|
| `scenario_depth` | 標的、方向、價格範圍、時點與資料／情境身分 |
| `participation_cap` | 明示的風控或研究上限，不冒充市場定律 |
| `capacity_qty` | 只對上述輸入成立的最大研究數量 |
| planned quantity | 與容量上限逐項比較 |
| 決定 | 超過上限時縮量或 no-go |

單一 order-book snapshot、bar volume 或固定參與率都不能單獨證明真實容量。若沒有
版本化深度、成交模型與規模敏感度，結果只能叫「容量情境」，不能叫「市場可容納
數量」。

## 最小決策表

| 證據 | 結論 |
|---|---|
| `EV_gross > 0`、`EV_net <= 0` | 成本情境下 no-go |
| `EV_net > 0`，但 planned quantity 超過壓力容量 | 規模 no-go；縮量後完整重算 |
| 只有勝率，沒有 gross／net outcome | 證據不足 |
| 成本已進 outcome，但又從 EV 扣一次 | 計算失敗；先修正雙扣 |
| 深度只是手填假設，卻標成真實容量 | 證據失實；降級為情境 |

## 適用邊界

- 本頁不推論未來勝率，也不提供策略獲利承諾。
- 期望值是輸入分布下的平均，不揭露 outcome 路徑、尾端損失或 risk of ruin。
- 容量與流動性會隨標的、時間、方向、數量及市場狀態改變。
- 回撤、波動率、Sharpe、Sortino、VaR 類指標與統計不確定性尚未在本 active slice
  展開。
- 真實費率、深度與市場資料必須在相符時點由第一手來源重新查證。
